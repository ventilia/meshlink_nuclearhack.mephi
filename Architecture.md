## 1. Общая идея: два слоя, чёткая граница

Весь проект делится ровно на два слоя с чёткой границей между ними.
```
╔══════════════════════════════════════════════════════════════════╗
║                                                                  ║
║   📱  ANDROID / KOTLIN СЛОЙ                                      ║
║   ─────────────────────────────────────────────────────────────  ║
║   Отвечает за:                                                   ║
║     • UI (Jetpack Compose, экраны, анимации)                     ║
║     • Android API (WiFi, BLE, Audio, Notifications)              ║
║     • База данных (Room), настройки (DataStore)                  ║
║     • Бизнес-логика на уровне приложения                         ║
║     • Жизненный цикл Android компонентов                        ║
║                                                                  ║
║   НЕ отвечает за:                                                ║
║     • Криптографию                                               ║
║     • Маршрутизацию пакетов                                      ║
║     • Идентичность узла                                          ║
║                                                                  ║
╠══════════════════════════╦═══════════════════════════════════════╣
║                          ║                                       ║
║    NativeCore.kt         ║   JNI граница                         ║
║    (       обёртка)      ║   (~10 нс на вызов)                   ║
║                          ║                                       ║
╠══════════════════════════╩═══════════════════════════════════════╣
║                                                                  ║
║   🦀  RUST ЯДРО (libmeshlink_core.so)                            ║
║   ─────────────────────────────────────────────────────────────  ║
║   Отвечает за:                                                   ║
║     • Генерацию и хранение криптографических ключей для           ║
                                                      ауиентификации║
║     • Идентичность узла (peer_id, short_code)                    ║
║     • Таблицу маршрутизации (RoutingTable + RouteMetrics)        ║
║     • Протокол сообщений (MessageType enum, сериализация)        ║
║     • Протокол файлов (чанки, хэши, retry)                       ║
║     • TCP соединения (ConnectionManager)                         ║
║     • Персистентность seed-ключа                                 ║
║                                                                  ║
║                                                  ║
║                                                   ║
║                                                             ║
║                                                   ║
║                                                                  ║
╚══════════════════════════════════════════════════════════════════╝
```

каждый слой меняется независимо. Поменяли UI — ядро не трогаем. Обновили алгоритм маршрутизации в Rust — компилируем только ядро, Android-код не меняется. Это стандартный принцип разделения ответственностей, применённый на уровне языков и рантаймов.

## 2. Rust-ядро: зачем и как

### Структура модулей

```
core/src/
│
├── jni_bridge.rs        ← публичный API, экспортируется в Android
│   │  Содержит ~20 JNI-функций. Каждая — тонкая обёртка над
│   │  внутренними модулями. Никакой логики прямо здесь.
│   │
│   ├── инициализация:  initWithFilesDir, generateOwnPeer
│   ├── идентичность:   getOwnPeerIdHex, getOwnShortCode, getOwnPublicKeyHex
│   ├── криптография:   signData, verifySignature
│   └── маршрутизация:  updateRoutingTable, getNextHopIp, getRoutingTableJson
│
├── crypto/
│   ├── keypair.rs       ← Ed25519: generate, from_seed, sign, verify
│   └── peer_id.rs       ← SHA-256(public_key) → hex string
│
├── peer/
│   └── peer.rs          ← структура Peer: id, name, short_code, public_key_bytes
│
├── protocol/
│   ├── message.rs       ← MessageType enum (14 вариантов), сериализация JSON
│   ├── codec.rs         ← encode/decode пакетов
│   └── file_transfer.rs ← OutgoingTransfer, IncomingTransfer, chunk logic
│
├── routing/
│   └── mod.rs           ← RoutingTable, Route, RouteMetrics + 6 тестов
│
├── connection/
│   ├── manager.rs       ← ConnectionManager: map peer_id → TCP stream
│   └── tcp.rs           ← низкоуровневый TCP
│
└── storage/
    └── mod.rs           ← load_seed / save_seed (Ed25519 private key на диске)
```

### Глобальное состояние — безопасно, потому что Rust

Всё состояние ядра хранится в статических `Lazy<Mutex<...>>` переменных:

```rust
static OWN_PEER:           Lazy<Mutex<Option<Peer>>>           = ...
static CONNECTION_MANAGER: Lazy<Mutex<ConnectionManager>>      = ...
static ROUTING_TABLE:      Lazy<Mutex<Option<RoutingTable>>>   = ...
static JAVA_VM:            Lazy<Mutex<Option<JavaVM>>>         = ...
static CALLBACK_OBJ:       Lazy<Mutex<Option<GlobalRef>>>      = ...
```

Это потокобезопасно **по гарантии компилятора**: Rust не позволит обратиться к `OWN_PEER` без блокировки мьютекса. Это принципиально отличает его от C++ (где можно забыть взять мьютекс и получить UB) и от Go (где компилятор не проверяет правильность синхронизации).

```
Поток JNI #1 (main thread)           Поток JNI #2 (network thread)
        │                                      │
        │ getOwnPeerIdHex()                    │ updateRoutingTable()
        │                                      │
        ▼                                      ▼
 OWN_PEER.lock()                      ROUTING_TABLE.lock()
        │                                      │
        │  ← разные мьютексы,                  │
        │    нет конкуренции                   │
        ▼                                      ▼
   peer.id.to_hex()              routing.learn_from_keepalive(...)
        │                                      │
 OWN_PEER.unlock()                ROUTING_TABLE.unlock()
```

Два потока обращаются к **разным** объектам — нет блокировок, нет гонок. Rust это гарантирует.

### Размер бинаря: ~300 KB

Настройки в `Cargo.toml` специально оптимизированы для мобильного:

```toml
[profile.release]
opt-level = "z"      # размер важнее скорости (для горячего пути не критично)
lto = true           # убирает весь неиспользуемый код между crates
codegen-units = 1    # лучше для LTO, медленнее компиляция
strip = true         # без debug-символов
panic = "abort"      # не тащим unwinding runtime (~50 KB экономии)
```

Для сравнения: аналогичная логика на Go дала бы `.so` размером 5–15 МБ из-за встроенного рантайма и GC.

## 3. JNI-мост: как Kotlin говорит с Rust

```
Kotlin (Android)                         Rust (Native)
─────────────────                        ─────────────
NativeCore.kt                            jni_bridge.rs
─────────────────                        ─────────────────────────────────

object NativeCore {
  init {
    System.loadLibrary(            →   [загрузка libmeshlink_core.so]
      "meshlink_core")
  }

  external fun                     →   #[no_mangle]
    getOwnPeerIdHex(): String           pub extern "system" fn
                                        Java_com_example_meshlink_core_
                                        NativeCore_getOwnPeerIdHex(
                                          env: JNIEnv, _class: JClass
                                        ) -> jstring {
                                          let hex = OWN_PEER.lock()
                                            .unwrap().as_ref()
                                            .map(|p| p.id.to_hex())
                                            .unwrap_or_default();
                                          env.new_string(hex).into_raw()
                                        }

  external fun
    signData(data: ByteArray)      →   Java_..._NativeCore_signData(
      : ByteArray                        mut env: JNIEnv,
                                         data: JByteArray
                                       ) -> jbyteArray { ... }

  external fun
    updateRoutingTable(            →   Java_..._updateRoutingTable(
      fromPeerId: String,                from_peer_id: JString,
      fromIp: String,                    from_ip: JString,
      peersJson: String                  peers_json: JString
    ): Int                             ) -> jint { ... }
}
```

Именование функций строго по соглашению JNI: `Java_{пакет}_{класс}_{метод}`. Rust компилятор проверяет типы на своей стороне, а `#[no_mangle]` гарантирует правильное имя в символах `.so`.

**Накладные расходы:** один JNI-вызов добавляет ~10–50 нс. Для криптографии (которая занимает микросекунды) это незначимо. Для маршрутизации (вызывается раз в несколько секунд при keepalive) — тем более.

## 4. Сетевой стек: многоуровневое обнаружение

Вместо единственного способа найти соседей, MeshLink использует **4 параллельных механизма**, каждый со своей нишей:

```
                    УСТРОЙСТВО
                        │
           ┌────────────┼────────────┐
           │            │            │
           ▼            ▼            ▼
    ┌─────────────┐ ┌──────────┐ ┌──────────────┐
    │UdpDiscovery │ │WiFiDirect│ │ BleDiscovery │
    │  :8810      │ │  P2P     │ │  GATT        │
    └──────┬──────┘ └────┬─────┘ └──────┬───────┘
           │             │              │
    ┌──────▼─────────────▼──────────────▼───────┐
    │              PeerRegistry                  │
    │  map: peerId → { ip, username, lastSeen }  │
    └──────────────────────┬────────────────────┘
                           │
                    ┌──────▼───────┐
                    │ NetworkManager│
                    │ (координатор) │
                    └──────────────┘
```

### Когда что работает

```
Ситуация                          Активный механизм
────────────────────────────────  ─────────────────────────────────
Оба в одной Wi-Fi сети            UDP Broadcast → быстро, 5 сек
Нет общей точки доступа           Wi-Fi Direct P2P → без роутера
Далеко, нет Wi-Fi                 BLE → до 50 метров
Корпоративная сеть (нет broadcast) NSD/mDNS → работает через роутер
```

### UDP Announce — что именно рассылается

```
Каждые 5 секунд (активный режим) каждый узел рассылает:

BROADCAST  → 192.168.0.255:8810    (всем в сети)
BROADCAST  → 192.168.49.255:8810   (Wi-Fi Direct сеть)
UNICAST    → 192.168.0.1:8810      (известные соседи)
UNICAST    → 192.168.49.1:8810

Содержимое пакета (~239 байт):
┌────────────────────────────────────────────────────────────────┐
│  peerId (hex, 64 символа)  │  username (UTF-8)                 │
│  shortCode (4 символа)     │  publicKeyHex (64 символа)        │
│  ipAddress                 │  keepalive timestamp              │
└────────────────────────────────────────────────────────────────┘

Из реальных логов:
  V  UDP packet from 192.168.49.1 (239 bytes)
  D  Announce from=192.168.49.1 peer=6fd5739d own=0a9df321
  V  Updated peer 'SM-A536E' @ 192.168.49.1
```

Система не реагирует на собственные анонсы (self-reflection через вторичные интерфейсы):
```
  D  Skip self-announce from 192.168.49.117
     (self-reflection, ownIpSet=[192.168.49.117, 192.168.0.119])
```

### Адаптивный keepalive

```
Активность устройства        Интервал keepalive    Причина
──────────────────────────   ──────────────────    ─────────────────────
Экран включён, чат открыт    7 секунд              быстрое обнаружение
Экран включён, idle          30 секунд             экономия батареи
Экран выключен               60 секунд             минимальная нагрузка
```

## 5. Mesh-маршрутизация: как пакет доходит через несколько узлов

### Сценарий: 4 устройства, не все видят друг друга

```
Реальная топология:

   [A] ────────── [B] ────────── [C]
  0a9df321      6fd5739d       a1b2c3d4
  192.168.49.117  192.168.49.1   192.168.0.50

                               [C] ──── [D]
                                       ff001122
                                       192.168.0.200

A видит B напрямую.
B видит A и C напрямую.
C видит B и D напрямую.
A не видит C и D вообще.
```

### Как A узнаёт о C и D (через keepalive)

```
Шаг 1: C отправляет keepalive B
   C → B: keepalive { peers: [(C, "192.168.0.50", hop=1)] }

Шаг 2: B обновляет свою таблицу
   B.routing_table: C → direct("192.168.0.50", hop=1)

Шаг 3: B отправляет keepalive A (включает информацию о C)
   B → A: keepalive {
     peers: [
       (B, "192.168.49.1",  hop=1),  ← прямой сосед
       (C, "192.168.0.50",  hop=1),  ← C знает о себе
       (D, "192.168.0.200", hop=2),  ← C знает о D
     ]
   }

Шаг 4: A обновляет таблицу маршрутов
   A.routing_table:
     B → Route { next_hop: "192.168.49.1", hops: 1, via: None     }
     C → Route { next_hop: "192.168.49.1", hops: 2, via: Some("B") }
     D → Route { next_hop: "192.168.49.1", hops: 3, via: Some("B") }
```

### Как A отправляет сообщение D

```
A ──[TCP:8800]──▶ B  (next_hop для D = B)
   Packet { receiverId: D, ttl: 5, payload: "привет" }

B видит receiverId=D, это не он сам.
B ищет D в своей таблице → next_hop = C.
B пересылает:
B ──[TCP:8800]──▶ C
   Packet { receiverId: D, ttl: 4, payload: "привет" }  ← TTL уменьшился

C пересылает D:
C ──[TCP:8800]──▶ D
   Packet { receiverId: D, ttl: 3, payload: "привет" }

D получает пакет, видит receiverId = свой ID → обрабатывает.
```

### RouteMetrics — не просто hop count

Rust-ядро не выбирает маршрут только по количеству хопов. Используется **функция стоимости**:

```rust
pub fn cost(&self, hop_count: u8) -> u32 {
    let hop_cost  = (hop_count as u32) * 100;   // базовая стоимость
    let rtt_cost  = self.avg_rtt_ms / 10;        // медленный канал = дороже
    let loss_cost = (self.packet_loss_percent as u32) * 2; // потери = дороже
    let fail_cost = self.consecutive_failures * 50;        // нестабильность
    hop_cost + rtt_cost + loss_cost + fail_cost
}
```

Пример: маршрут через 2 хопа с 0% потерями предпочтительнее маршрута через 1 хоп с 30% потерями:
```
Маршрут A (2 хопа, 0% потери):  2*100 + 0 + 0 + 0 = 200
Маршрут B (1 хоп, 30% потери):  1*100 + 0 + 60 + 0 = 160  ← выбирается B

Маршрут A (2 хопа, 0% потери):  200
Маршрут B (1 хоп, 50% потери):  100 + 100 = 200  ← равноценно → остаём на A
```

### Защита от петель: два независимых механизма

```
Механизм 1: TTL
───────────────
Каждый пакет рождается с TTL=5.
Ретранслятор принимает пакет только если TTL > 0,
отправляет с TTL-1.
Максимальная глубина сети = 5 хопов = MAX_HOPS+1.

A(ttl=5) → B(ttl=4) → C(ttl=3) → D(ttl=2)
                                 ↓ TTL=0 → пакет удалён, не идёт дальше

Механизм 2: PacketDeduplicator
───────────────────────────────
Каждый узел хранит LRU-кэш (messageId → timestamp).
Если пакет уже был обработан — тихо отбрасывается.

Это защищает от колец: A→B→C→A→...
Даже если топология имеет петли, пакет не зациклится.

Лог при дублировании:
  D  🔄 Дубль отброшен | id=1741376221478
```

## 6. Голосовые звонки: почему UDP и как работает буфер

### Почему не TCP для голоса

```
TCP — гарантия доставки:              UDP — нет гарантии:
────────────────────────              ───────────────────
A → B: [пакет 1]           OK        A → B: [пакет 1]   OK
A → B: [пакет 2]           LOST      A → B: [пакет 2]   LOST
A → B: [пакет 2, retry]    300мс!    A → B: [пакет 3]   OK  (пропустили 2)
A → B: [пакет 3]           ждём...

При TCP: если пакет потерян, следующие ждут retransmit.
Задержка может вырасти до 200–500 мс. Голос «заикается».

При UDP: потерянный пакет просто пропускается.
20 мс тишины почти не слышно. Следующий пакет идёт сразу.
```

### Формат UDP-пакета звонка

```
 0       1       2       3       4       5
 ┌───────┬───────┬───────┬───────┬───────┬───────┬─────────────────┐
 │  seq  │  seq  │ flags │  rtt  │  rtt  │ spare │   PCM data...   │
 │ [0]   │ [1]   │       │ [0]   │ [1]   │       │   (переменно)   │
 └───────┴───────┴───────┴───────┴───────┴───────┴─────────────────┘
   ◄──── 6 байт заголовок ───────►◄──── PCM 16-bit mono ───────────►

flags:
  0x01 = FLAG_PING   (запрос RTT)
  0x02 = FLAG_PONG   (ответ RTT)
  0x04 = FLAG_MUTED  (микрофон выключен, не воспроизводить)
```

### Адаптивный jitter-буфер

Джиттер — это разброс времени прихода пакетов. Если пакеты приходят нерегулярно, звук «скачет». Буфер накапливает несколько пакетов перед воспроизведением, сглаживая неравномерность.

```
Измеренный джиттер        Размер буфера     Задержка воспроизведения
─────────────────────────  ───────────────   ────────────────────────
< 30 мс (стабильный канал)  3 пакета          ~60 мс
30–80 мс (нормальный)        5 пакетов (def)   ~100 мс
> 80 мс (нестабильный)       12 пакетов (max)  ~240 мс

Из реальных измерений на Wi-Fi Direct:
  V  Метрики: RTT=57мс потери=6,9% jitter=98мс буфер=12 мут=false
  V  Метрики: RTT=57мс потери=5,1% jitter=83мс буфер=12 мут=false

RTT=57мс → one-way latency ≈ 28мс
С буфером 12 пакетов по 20мс = +240мс
Итого one-way: ~268мс — в пределах комфортного разговора (< 400мс)
```

### Полный сигнальный путь звонка

```
Инициатор (A)                              Получатель (B)
     │                                          │
     │  TCP: CALL_REQUEST                       │
     │  { senderId, receiverId, callType=AUDIO }│
     │─────────────────────────────────────────▶│
     │                                          │  → уведомление
     │                                          │  → GlobalCallManager
     │                                          │    .startRingtone()
     │  TCP: CALL_RESPONSE { accepted: true }   │
     │◀─────────────────────────────────────────│
     │  stopRingtone()                          │  stopRingtone()
     │                                          │
     │  AudioRecord.startRecording()            │  AudioRecord.startRecording()
     │  CallManager.startSession(B.ip)          │  CallManager.startSession(A.ip)
     │                                          │
     │◄══════════ UDP :8813 ════════════════════▶│
     │  [seq][flags][rtt][spare][PCM 16-bit]    │
     │  каждые ~20 мс                            │
     │                                          │
     │  PING(FLAG_PING) ───────────────────────▶│
     │◀──────────────────────── PONG(FLAG_PONG) │
     │  RTT = now - ping_sent_ms                │
     │  (каждые 2 секунды)                      │
     │                                          │
     │  TCP: CALL_END                           │
     │─────────────────────────────────────────▶│
     │  stopSession()                           │  stopSession()
     │  AudioRecord.stop()                      │  AudioRecord.stop()
```

## 7. Передача файлов: надёжность без блокировки сети

### Почему не просто один большой пакет

Прямая передача файла в одном пакете:
- Нет контроля целостности: один битовый сбой = весь файл заново
- Нет возобновления: разрыв = начинай с нуля
- Блокирует канал: другие сообщения ждут

Chunked-протокол решает всё это:

```
Файл 1 МБ → 128 чанков по 8 КБ
                │
    ┌───────────┼───────────┐
    ▼           ▼           ▼
  чанк 0     чанк 1     чанк 2   ...
  SHA-256    SHA-256    SHA-256
    │           │           │
  ACK/RETRY  ACK/RETRY  ACK/RETRY
    │
  Параллельно: текстовые сообщения идут в тех же TCP-соединениях
  без задержки (приоритет через очередь)
```

### Протокол с recovery

```
Отправитель A                    Получатель B
      │                               │
      │── FILE_INIT ────────────────▶ │  transferId, totalChunks=128,
      │   { fileHash: sha256(file) }  │  fileSize, mimeType
      │                               │
      │── CHUNK #0 ────────────────▶  │  data=8KB, chunkHash=sha256(data)
      │◀─ CHUNK_ACK #0 ───────────────│  receivedHash = sha256(полученного)
      │   ✅ хэши совпали             │
      │                               │
      │── CHUNK #1 ────────────────▶  │
      │   [разрыв соединения]         │
      │                               │
      │   [переподключение через 30с] │
      │                               │
      │── FILE_STATUS_REQUEST ───────▶│
      │◀─ FILE_STATUS_RESPONSE ───────│  received: [0], missing: [1..127]
      │                               │
      │── CHUNK #1 [resume] ────────▶ │  продолжаем с #1, не с начала
      │── CHUNK #2 ─────────────────▶ │
      │   ...                         │
      │── FILE_COMPLETE ────────────▶ │
      │   { finalHash: sha256(file) } │  проверяем итоговый хэш
      │                               │  ✅ совпал → файл принят
```

### Ограничение нагрузки

```
Очередь передач:           [файл1] → [файл2] → [файл3]
                                │
                           только один активный transfer

Окно ACK:                  чанк N отправляется только после ACK(N-1)
                           нет "flood" чанков без подтверждения

Coroutine scope:           Dispatchers.IO — не блокирует Main thread
                           UI остаётся отзывчивым во время передачи
```

## 8. Безопасность: идентичность через криптографию

### Как рождается идентичность узла

```
Первый запуск:

OsRng (крипто-безопасный)
    │
    ▼
Ed25519 SigningKey::generate()
    │
    ├── signing_key (32 байта)  ──▶  storage::save_seed(filesDir)
    │   [НИКОГДА не покидает устройство]     (защищено sandbox Android)
    │
    └── verifying_key (32 байта)  = public_key
            │
            ▼
        SHA-256(public_key)
            │
            ▼
        peer_id = hex(sha256)[0..32]   = "0a9df3218f4bc..."
        short_code = peer_id[0..4]     = "0a9d"  ← показывается в UI

Следующие запуски:

storage::load_seed(filesDir)
    │
    ▼
Ed25519 SigningKey::from_seed(seed)  ← детерминировано
    │
    └── тот же peer_id, та же идентичность
```

### Что гарантирует эта схема

```
Свойство                        Как обеспечивается
──────────────────────────────  ────────────────────────────────────
Уникальность peer_id            SHA-256(Ed25519 public key) — коллизии
                                невозможны на практике

Невозможность подделать peer_id Нельзя вычислить signing_key из peer_id
                                (односторонняя функция SHA-256)

Аутентификация при Handshake    A отправляет public_key.
                                B может верифицировать подпись A:
                                verify(A.public_key, data, signature)

Персистентность идентичности    seed хранится в filesDir — только
                                наше приложение имеет доступ без root
```

### Защита от атак в сети

```
Атака              Механизм защиты
──────────────────  ──────────────────────────────────────────────
Replay attack       PacketDeduplicator: LRU-кэш messageId.
                    Повторный пакет отбрасывается немедленно.
                    Лог: D  🔄 Дубль отброшен | id=1741376221478

Flood / DoS         Rate limiting: max N пакетов/сек от каждого peer_id.
                    Лог: W  🚫 Rate limit | a3f9d21b...

Loop attack         TTL на каждом пакете (max 5 хопов).
                    + PacketDeduplicator как второй барьер.

Identity spoofing   peer_id = SHA-256(public_key).
                    Без приватного ключа нельзя подписать данные
                    от имени другого peer_id.

Self-announce loop  Фильтрация по IP + peer_id при обработке UDP.
                    Лог: D  Skip self-announce from 192.168.49.117
```

## 9. Android-слой: архитектура приложения

### Почему нет Hilt/Dagger

DI сделан вручную через `AppContainer` + `AppDataContainer`. Это осознанный выбор:

```
Hilt/Dagger:                         AppContainer (наш подход):
────────────────────                 ────────────────────────────
+ автогенерация кода                 + нет зависимости от annotaion processor
- +2–4 МБ в APK                      + нет холодного старта из-за reflection
- медленный первый старт             + полная прозрачность: видно что и когда
- сложная отладка                      создаётся
- не нужен для нашего масштаба       + легко тестировать (передать mock)
```

### Жизненный цикл: порядок инициализации важен

```
Время ──────────────────────────────────────────────────────────────▶

T=0  Application.onCreate()
     └── MeshLinkAppProvider.app = this   ← глобальная ссылка на Application

T=1  MainActivity.onCreate()
     ├── app.initializeContainer(this)    ← создаём Room DB, репозитории
     │   Нельзя раньше: нужен Context Activity для некоторых DAO
     ├── NotificationHelper.createChannels()
     └── requestPermissions()

T=2  [Пользователь даёт разрешения]

T=3  startForegroundService(WiFiDirectService)

T=4  WiFiDirectService.initialize()
     ├── nm = container.networkManager
     ├── app.networkManager = nm          ← сохраняем ссылку в Application
     ├── nm.start()
     │   ├── NativeCore.initWithFilesDir() ← Rust инициализирует ключи
     │   ├── MeshServer.start() :8800
     │   ├── UdpDiscovery.start() :8810
     │   └── BleDiscovery.start()
     └── app.notifyServiceContainerReady()
             └── GlobalCallManager.attach(nm)  ← рингтон теперь работает

ВАЖНО: notifyServiceContainerReady() вызывается ПОСЛЕ nm.start(),
       потому что GlobalCallManager.attach() требует уже запущенного NM.
       В MainActivity это вызывать нельзя — nm ещё null.
```

### GlobalCallManager — зачем он нужен

Без него проблема: рингтон запускался только внутри `ChatViewModel` конкретного пира. Если пользователь находился на главном экране или в настройках — звонок пропускался.

```
Старое поведение:                    С GlobalCallManager:
──────────────────────────           ────────────────────────────────
ChatScreen (открыт чат с A)          Любой экран
    │                                     │
    │ callRequest от A                    │ callRequest от A
    ▼                                     ▼
 рингтон играет ✅                    GlobalCallManager (Application scope)
                                          │
ChatListScreen (список чатов)             ▼ collectLatest(nm.callRequest)
    │                                 рингтон играет ✅ ВСЕГДА
    │ callRequest от B
    ▼
 рингтон НЕ играет ❌
```

GlobalCallManager живёт на уровне `Application` — он активен всегда, пока процесс жив.

### Адаптивное поведение сервиса

```
Событие                    Действие
─────────────────────────  ──────────────────────────────────────
MainActivity.onResume()    AppForegroundTracker.setForeground(true)
                           → discovery активна, keepalive 7 сек

MainActivity.onPause()     AppForegroundTracker.setForeground(false)
                           → discovery приостановлена

Экран выключен             networkManager.pauseDiscovery()
                           keepalive → 60 сек

Входящее сообщение         NotificationHelper.showMessageNotification()
(app в фоне)               → уведомление с текстом

Входящий звонок            NotificationHelper.showCallNotification()
(app в фоне)               → fullscreen intent, category=CALL
                           → GlobalCallManager.startRingtone()
```

## 10. Почему эта архитектура хооршая и особенности

### 1. Критический код изолирован в Rust

Криптография, маршрутизация, протокол файлов — всё, что сложно и важно правильно реализовать — написано на языке, который **не позволяет ошибиться** 

### 2. Многоуровневый fallback транспорта

 есть Wi-Fi — используем UDP, нет — Wi-Fi Direct, нет и его — BLE. Связь не прерывается при смене условий.

### 3. Маршрутизация с метриками качества

Простой алгоритм (минимум хопов). Функция стоимости учитывает RTT, процент потерь и количество сбоев система сама выбирает лучший путь.

### 4. Адаптивный jitter-буфер вместо фиксированного

Фиксированный буфер — компромисс: либо маленький (экономим задержку, но звук "дёргается"), либо большой (задержка растёт). Адаптивный буфер даёт минимальную задержку на хорошем канале и автоматически увеличивается при деградации сети.

### 5. Двойная защита от петель в mesh

TTL + PacketDeduplicator — это два независимых барьера. Если TTL по какой-то причине не сработал (баг в relay-логике), дедупликатор остановит пакет. Независимые механизмы защиты — стандарт надёжных протоколов.
